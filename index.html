We can see that measuring total changes made in commits to a repository is not a good way to measure the productivity of a software engineer by simply looking at the grah. The massive spike at date 2021/12/07 does not mean i was particularly productive in that commit, i simply scrapped the use of a javascript framework for this visualisation which involved the deletion of many files and in conjunction many lines of code. Even if i delete commits that are larger than three thousand changes we still cannot use this flawed metric to measure software engineers.
Here we can see that a similar spike is alswe present where files have been added or deleted
From the data gathered that was used to create this visualisation we can see that total changed lines of code cannot be used as a reliable metric to measure software engineers performance mainly because of the point mentioned above, i go into why lines of code cannot be used as a way to measure a single software engineers performace in more detail in my report Total lines of code changed however can be usefull to visualise the progress of the project, if we see a spike in the graph that is an outlier among other commits we could come to the conclusion that a whole part of the project has been scrapped or code has been reused from other projects, with some research we can figure out that maybe there has been problems in the development. for example from the first graph above you can see that in one commit i changed nearly two and a half thousands lines of code, this could indicate a change of tool or approaches used to complete the project, this is true in this example as i dropped the use of react in favour of vanilla javascript and html
